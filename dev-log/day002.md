# Programming Concepts Learned

Today I learned that:

- Boolean variables store either `True` or `False`.
- Variables only store the value assigned at that moment.
- Variables must be updated whenever the game state changes.
- Code placement inside loops is just as important as the code itself.
- Understanding execution order is essential for debugging.
- Good debugging comes from tracing execution rather than randomly changing code.
- Algorithm design and optimisation are separate skills from writing Python syntax.

# Challenges

## Challenge 1 - Detecting Dead Pumpkins

### Problem

Initially I struggled to detect dead pumpkins correctly.

I experimented with dictionaries before realising the issue wasn't storing information—it was understanding when variables update.

### Solution

I moved the dead pumpkin detection inside the traversal loop so it was recalculated for every tile.

```python
is_dead_pumpkin = get_entity_type() == Entities.Dead_Pumpkin
```

### Lesson Learned

Variables represent a snapshot of the current state and do not automatically update as the farmer moves.

## Challenge 2 - Understanding Execution Flow

### Problem

Even after fixing the detection logic, the farmer still behaved unexpectedly.

### Solution

Instead of randomly changing code, I traced the farmer's actions tile by tile to understand exactly what the program was doing.

### Lesson Learned

Understanding execution flow is one of the most valuable debugging skills.

## Challenge 3 - Mega Pumpkin Optimisation

### Problem

Harvesting pumpkins immediately prevented large Mega Pumpkins from forming.

I wanted to maximise yield while automatically replacing dead pumpkins.

### Solution

I designed and tested several different repair strategies, eventually settling on a fixed repair scan algorithm as a temporary solution while planning a more adaptive approach.

### Lesson Learned

Sometimes a simple working solution is the best foundation before attempting further optimisation.

---

## Challenge 4 - Comparing Theory Against Reality

### Problem

My repair algorithm wasn't behaving exactly as expected.

Initially I believed my implementation matched my mathematical model.

### Solution

After reviewing both my code and my recorded test data, I realised I had accidentally counted the initial planting traversal as one of the repair scans.

This meant my implementation was performing fewer repair scans than intended.

### Lesson Learned

Always verify that the implementation actually matches the algorithm you designed.
