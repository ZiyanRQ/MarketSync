# Programming Concepts Learned

Today I learned that:

- `return` inside a loop exits the function immediately — it doesn't let the loop finish iterating.
- You can't perform arithmetic directly between incompatible types (like a list and an int).
- Dictionaries don't have list methods like `.append()` or `.index()` — the two data structures behave very differently.
- `dict.items()` lets you loop through key–value pairs together, which is often more useful than looping through keys or values alone.
- The left-hand side of an assignment (`=`) must be a variable or valid reference — not a function call or an expression like `thing['address']`.
- In Python, defining the same method name twice in a class silently overwrites the first definition — they don't merge or overload.
- A variable can't be used to calculate its own value before it's been defined (no self-referencing on first assignment).
- `super().__init__()` lets a subclass reuse a parent class's constructor logic instead of duplicating it.
- Subclasses don't need to override methods (like `area()`) if the parent class's existing logic already works correctly once the right attributes are set.

# Challenges

## Challenge 1 - Updating Dictionary Values

### Problem

Tried to add a number to every value in a dictionary, but the function returned early and attempted invalid arithmetic between a list and an integer.

### Solution

Learned to loop through `.items()`, build a **new** dictionary, and add `n` to each individual value rather than trying to operate on the whole list of values at once.

### Lesson Learned

Operations that work on individual numbers don't automatically apply across an entire list — you need to loop and act on each element.

---

## Challenge 2 - Finding Students by Address

### Problem

Initial attempts confused dictionary and list methods (`.index()`, `.append()` on the wrong types), used invalid loop syntax, and referenced undefined variables (`student` vs `thing`).

### Solution

Worked through the logic step by step: loop through the dictionary's values, compare each student's address to the target address with a plain `if` statement, and append matching names to a list.

### Lesson Learned

A single loop with a clear `if` condition is often all that's needed — nested loops or extra method calls usually mean the problem hasn't been fully thought through yet.

---

## Challenge 3 - Square Subclass and `super()`

### Problem

Multiple attempts redefined `__init__` more than once (silently overwriting earlier versions), referenced undefined variables (`x2`, `y2`) before calculating them, and mixed up which coordinate belonged to which calculation.

### Solution

Settled on a single `__init__(self, x1, y1, length)` method that calculates `x2` and `y2` from `x1`, `y1`, and `length` **before** calling `super().__init__(x1, y1, x2, y2)`, then removed the unnecessary custom `area()` method since `Rectangle` already provides one.

### Lesson Learned

Order of operations matters — a variable must be fully defined before it's used, and inheritance means you often need fewer methods, not more.

# Biggest Lessons

- Debugging is often about spotting undefined variables and mismatched types before the code even runs.
- Redefining a method with the same name erases the previous one — Python doesn't merge them.
- Inheritance is most powerful when it removes duplicate code, not when it's used to write parallel versions of existing logic.
- Working through problems interactively, one small fix at a time, surfaces more bugs than trying to fix everything at once.
