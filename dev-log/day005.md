# The Project: Recap & Direction

## Final Goal

Build a **market screener** in Python that compares YES/NO prices for matching events across **Kalshi** and **Polymarket**, and flags price discrepancies where combined cost (YES on one side + NO on the other) is under $1 — a guaranteed arbitrage spread before fees.

Example: YES at 60¢ on one platform + NO at 30¢ on the other = 90¢ total cost, guaranteed $1 payout = 10¢ profit minus fees.

## Scope Note

This project is scoped as a **screener only** — identifying and displaying price discrepancies, not acting on them. I'm UK-based, and jurisdictional/eligibility restrictions may apply to actually trading on these platforms, so I'm deliberately keeping the project to data comparison and analysis for now rather than building anything that places trades.

## Explicitly Out of Scope

- Automated cross-platform market matching (too complex to solve alongside the arbitrage logic itself — matching will be done manually/hardcoded for now).

## Key Technical Findings From Research

- **Kalshi**: all market data (prices, order books, market metadata) is publicly available without authentication. Auth is only needed later, for placing trades.
- **Polymarket**: split into separate APIs — Gamma (public, market discovery/metadata) and CLOB (order book + trading, auth required only for placing orders). For a screener, I only need Gamma.
- **Practical implication**: no account verification or API keys are needed to start building the screener. I can start hitting public endpoints on both platforms immediately once my Python fundamentals are solid enough.

## Market Matching Strategy

Rather than trying to auto-match questions by text/keywords (a hard NLP problem on its own), I'll:

1. Manually pick 5–10 markets where I can visually confirm both platforms are tracking the exact same resolution event (same date, same source, same threshold).
2. Hardcode the mapping (e.g. Kalshi ticker ↔ Polymarket condition ID) in a small config/dict.
3. Build and test the screener logic against that fixed list first.
4. Treat automated market-matching as a possible future stretch goal, not a current requirement.

**Priority — genuinely new material:**

- **Classes** — never covered. Relevant for structuring screener data cleanly (e.g. a `Market` or `PriceQuote` class) instead of loose dictionaries.
- **Asynchronous Programming** — higher priority than it looks. The screener needs to fetch prices from two platforms; doing this synchronously risks price drift between reads. Async lets both fetches happen concurrently.
- Iterators / Generators — lower priority, useful for cleaner data processing later.

**Planned order:** Iteration & Loops → Classes → Asynchronous Programming → (later) Iterators/Generators/Dictionaries-skim.
