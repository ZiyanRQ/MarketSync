I also installed the Jupyter extension so I could run notebook-style `.ipynb` files with output appearing directly beneath each code cell, which is closer to how DataCamp displayed results.

For running plain `.py` scripts I installed the Code Runner extension, which adds a one-click "Run" option. By default its output was cluttered with previous runs, extra language configs I didn't need, and it wasn't running inside the actual terminal panel — so I edited its settings so it clears old output on each run, runs the whole file (not just a highlighted selection), and executes inside the integrated terminal rather than a separate output tab. I also set Git Bash as my default integrated terminal and changed the editor theme to make the workspace feel cleaner and more focused.

While configuring this, I learned the difference between VS Code's **global** settings (`settings.json` under User Settings, applies everywhere) and **workspace** settings (`.vscode/settings.json`, applies only to the current project folder). I initially made some of these changes globally by mistake, then reverted them by identifying the exact lines I'd added (`terminal.integrated.defaultProfile.windows`, the `python` entry inside `code-runner.executorMap`, and `code-runner.runInTerminal`) and removing them, restoring my global settings to just the original theme and autosave preferences. I then re-added the same configuration at the workspace level instead, so it only affects this Educative project:

```jsonc
{
    "terminal.integrated.defaultProfile.windows": "Git Bash",
    "code-runner.runInTerminal": true,
    "code-runner.clearPreviousOutput": true,
    "code-runner.ignoreSelection": true,
    "code-runner.executorMap": {
        "python": "clear && python -u"
    }
}
```

I later added a further block of workspace settings to disable autocomplete, hover tooltips, parameter hints, auto-closing brackets/quotes, and inline suggestions — since for learning purposes I wanted to type exercises out myself without VS Code suggesting or auto-completing code for me.

# Programming Concepts Learned

Today I learned that:

- `for` loops are best when the number of iterations is known upfront (e.g. looping over a list or a fixed range).
- `while` loops are best when looping depends on a condition that changes during execution (e.g. counting down until a value hits zero).
- Nested loops let you compare every element in a list against every other element.
- Loop **order of operations** matters — checking/printing before modifying a variable gives a different result than modifying before checking/printing.
- Off-by-one errors are one of the most common loop bugs (wrong `range()` start/stop, or `i - 1` / `i + 1` running out of bounds).
- Negative indexing (`lst[-1]`) silently returns a valid value instead of erroring, which can hide bugs rather than surface them.
- Workspace-level settings in VS Code override global settings, and are useful for keeping project-specific configuration isolated.

## Challenge 3 - Detecting Duplicate Numbers

### Problem

My first attempt used nested loops but had two bugs: comparing each number to itself (`range(i, len(lst))` instead of `range(i + 1, len(lst))`), and returning `False` inside the loop on the first non-match instead of after checking every pair.

### Solution

Fixed the inner loop to start at `i + 1` (avoids self-comparison and duplicate pair-checks), and moved `return False` outside both loops so it only runs after every pair has been checked.

```python
def hasDuplicates(lst):
    for i in range(len(lst)):
        for j in range(i + 1, len(lst)):
            if lst[i] == lst[j]:
                return True
    return False
```

### Lesson Learned

`return False` placed inside a loop body tied to a single comparison will exit early and give false negatives — it needs to be reserved for "checked everything, found nothing."

## Challenge 4 - Even/Odd Countdown

### Problem

Needed to print a number, label it even/odd, decrease it, and repeat down to 1. My early attempts either counted the wrong direction, skipped the original input value, or decreased the number before printing it.

### Solution

Solved it two ways — once with a `for` loop counting down via `range(n)` and decrementing after printing, and once with a `while (n > 0)` loop, which handled the stopping condition more naturally.

```python
def printEvenOdd(n):
    while n > 0:
        if n % 2 == 0:
            print("Even number:", n)
        else:
            print("Odd number:", n)
        n -= 1
```

### Lesson Learned

When the number of loop iterations depends on a changing value rather than a fixed count, a `while` loop expresses the intent more directly than calculating a `range()`.
